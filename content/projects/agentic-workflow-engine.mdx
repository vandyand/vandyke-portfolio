## Problem

Most "AI agent" glue code is a pile of ad-hoc `fetch` calls and prompt
strings with no contract between steps. When step three returns something
step four didn't expect, you find out in production. I wanted workflow
orchestration where the contract *is* the architecture: every action
declares its input and output schema, and the engine enforces the wiring.

## What I built

A workflow orchestration engine where workflows are data, not code:

- **Typed action registry** — each action (HTTP GET, XML→JSON transform,
  JQ-style data transform, LLM call) declares JSON-Schema inputs and
  outputs. A workflow is a set of named nodes referencing actions.
- **DAG execution** — the runner resolves node dependencies via
  topological ordering, so steps run as soon as their inputs exist. No
  hand-sequenced call chains.
- **Error handling as a first-class feature** — per-node retries,
  timeouts, and graceful fallbacks are declared in the workflow
  definition, and one of the bundled demos exists purely to show a retry
  recovering from a failing endpoint.
- **Observability built in** — the UI streams per-node execution details:
  status, timing, raw output for every step, plus a run-level summary
  (nodes executed, success ratio, total time) and the raw execution log.

The live demo ships real workflows against real APIs — an arXiv paper
search (HTTP GET → XML parse) and a Wikipedia summary lookup — with the
DAG visualized alongside the execution results.

## Outcome

Deterministic, inspectable multi-step automation: pick a workflow, run
it, and read exactly what happened at every node. The engine pattern —
schema contracts between steps, dependency-ordered execution, declared
failure policies — is the same one I bring to client agent pipelines,
scaled down to a demo you can poke at in the browser.
